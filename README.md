A lightweight, secure, and production-ready HTTP server blueprint built in Go (1.22+). This project demonstrates enterprise-grade architecture featuring structured logging with log/slog, secure JSON decoding with size constraints, clean routing, and HTTP state-tracking middleware. Key FeaturesStructured Logging (log/slog): Outputs logs in text-based key=value format (logfmt), perfect for high-speed analysis and compatibility with aggregators (Grafana Loki, Datadog).HTTP State Tracking: Custom LoggingResponseWriter captures HTTP status codes and response sizes, overcoming the limitations of Go's standard library.Global Middleware: Measures and logs every request's execution time, path, HTTP method, and client IP.Defensive JSON Parsing: Uses stream-based json.NewDecoder paired with DisallowUnknownFields() to prevent data poisoning.DoS Protection: Request body size limits (http.MaxBytesReader) and strict server timeouts shield the application from memory exhaustion and slow-network attacks. Architecture & Components Breakdown1. Logging Layer (initLogger)Configures a centralized *slog.Logger utilizing NewTextHandler. It forces development-friendly text output, unlocks the Debug visibility level, and injects immutable global metadata into every log frame:goslog.String("env", "local"),
slog.String("version", "1.0.0"),
.2. HTTP State Capture (LoggingResponseWriter)Standard Go http.ResponseWriter is write-only regarding status codes. This structural wrapper intercepts the transmission pipeline via custom Write and WriteHeader methods to preserve transaction metadata (statusCode and written bytes) for the logging middleware.3. Middleware Pipeline (loggingMiddleware)Interceptors wrapped around the primary router (http.NewServeMux). It calculates overall latency via time.Since(start) and handles runtime statistics generation.4. Payload Interception (handleCreateComment)Extracts dynamic parameters via native path routing (r.PathValue). It enforces a strict 1 Megabyte maximum payload cap and immediately drops unexpected JSON keys, keeping your storage payload predictive and secure. Pros and Cons (Pros & Cons)Advantages (Pros)Zero External Dependencies: Built entirely using the Go standard library, keeping the compilation artifact small and maintaining optimal execution speed.Extremely Secure: Hardened out of the box against common vectors like Slowloris, large header floods, and unmetered memory injection attacks.Excellent Observability: Structured logging means you can easily parse and search through logs using modern telemetry backends without writing complex regex scripts.High Efficiency: Utilizing json.NewDecoder reads network data streams progressively, reducing spikes in RAM usage compared to json.Unmarshal.Disadvantages (Cons)Log Incompressibility for Complex Types: While key=value text representation is easy to read for humans during local development, it struggles to elegantly depict deeply nested structural objects (JSON format is preferred for heavy nested tracking).Manual Scalability: Since it avoids monolithic third-party web frameworks (like Gin or Fiber), routing groups, automated input validation, and object relational mapping must be structured manually as the application expands.Static Context Invalidation: Assigning the logger to a global var logger variable limits your ability to safely mutate logging context across asynchronous goroutine boundaries (passing the logger via context.Context is better for larger systems).🏎️ Quick Start1.
